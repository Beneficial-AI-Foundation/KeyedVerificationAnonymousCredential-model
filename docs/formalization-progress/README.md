# Formalization progress

­¤ōŖ **[View the current progress table ŌåÆ](FORMALIZATION_PROGRESS.md)**

A small, dependency-light tool that maps the elements of the source paper
(Orr├╣, *Revisiting Keyed-Verification Anonymous Credentials*, IACR ePrint
2024/1552) to the Lean declarations that formalize them, and renders a progress
table. It answers, per paper definition/lemma/theorem (and selected
figures/sections): *is this formalized in Lean yet, by which declaration, and is
the proof complete?*

It is a generated, drift-checked companion to the hand-maintained `## Status`
board in the top-level `README.md`. That board tracks **work items**; this tool
tracks **paper elements** and derives everything from the sources, so it cannot
silently disagree with the code.

## How it works

1. **Paper side.** The committed text extraction of the paper
   (`Orru_2024.txt`, written once by the `extract` subcommand via
   `pdftotext -layout`) supplies the numbered environments
   (Theorem/Lemma/Definition/Claim/Corollary), figure captions, and section
   headings, recording each element's enclosing section and PDF page.
   `report` and `--check` read this file, never the PDF, so their output does
   not depend on the local poppler version.
2. **Lean side.** Every `*.lean` under `lean_root` (minus git-ignored paths and
   `exclude_dirs`) is scanned for declarations (`def`, `structure`, `abbrev`,
   `class`, `instance`, `inductive`, `lemma`, `theorem`) and the paper citations
   attached to each ŌĆö in its `/-- ŌĆ” -/` docstring and signature, or its module
   `/-! ŌĆ” -/` doc.
3. **Join.** The two sides meet on a canonical reference key. The link is the
   **citation convention**: Lean docstrings cite the paper with the tag `O24`
   followed by the element, e.g.

   ```
   /-- ŌĆ” per O24 Definition 3.1 ŌĆ” -/
   structure AlgebraicMACSyntax ŌĆ”
   ```

   associates that structure with the paper's Definition 3.1. Recognized forms:
   `O24 Definition 3.1`, `O24 Theorem 5.1`, `O24 Figure 5`, `O24 ┬¦3.3`. The tag
   prefix is optional when matching, but using it keeps citations unambiguous.

### Status of a row

| Mark | Meaning |
|---|---|
| ­¤¤ó | a `sorry`-free Lean declaration of the **matching kind** is cited |
| ­¤īÉ | a matching declaration exists but its body contains `sorry`/`admit` |
| ­¤īĆ | a declaration is cited but its kind does not match the paper object |
| ­¤¤Ī | only module-level (`/-! ŌĆ” -/`) coverage |
| ŌÜ¬ | nothing yet |

"Matching kind" (the paper's *signature* for the object): a Definition,
Construction, Figure (game/construction) or Section (assumption/interface) must
be backed by a definitional construct; a Theorem/Lemma/Claim/Corollary must be
backed by a `theorem`/`lemma`.

### Figures and sections

Numbered environments are always rows. Figures and sections appear **only if
they are curated** in `element_summaries.toml`, i.e. only when they carry a
mathematical object that should be formalized and is part of the paper's
contributions (a security game, a construction, a hardness assumption, a
proof-system interface).

## Setup

No Python packages are needed; the script is standard-library only. It requires:

- **Python Ōēź 3.11** (for the built-in `tomllib`).
- **pdftotext** (poppler-utils) ŌĆö only for the `extract` and `init`
  subcommands, which run once per paper; `report` and `--check` read the
  committed extraction and need no poppler.
- **git** (optional) ŌĆö when present, Lean files matching `.gitignore` or
  `.git/info/exclude` are skipped.

On Debian/Ubuntu:

```bash
sudo apt-get install -y poppler-utils git
```

See `requirements.txt` for the same, in machine-readable form.

## Command-line use

Run from the repository root. Let `P=docs/formalization-progress/formalization_progress.py`.

### Generate / refresh the source descriptor (`init`)

`init` writes `formalization_source.toml` by extracting metadata from a paper
file and/or a URL. Provide at least one of `--pdf` / `--url` (they are
complementary):

```bash
python3 $P init --pdf docs/Orru_2024.pdf --url https://eprint.iacr.org/2024/1552
```

- The **title** comes from the URL (when reachable) or the PDF.
- The **citation tag** is detected from the existing Lean citations (`O24`
  here), falling back to author-initials + year; override with `--tag`.
- With only `--url`, the PDF is downloaded when possible (eprint/arXiv).

### Extract the paper text (`extract`, once per paper)

```bash
python3 $P extract --config docs/formalization-progress/formalization_source.toml
```

Runs `pdftotext -layout` on the source PDF and writes the committed extraction
(the `text` field of the source descriptor, `Orru_2024.txt` here). This is the
only step that needs poppler. Commit the result; rerun only if the PDF or the
extraction tooling changes.

### Generate the progress table (`report`, the default)

```bash
python3 $P                                           # uses the built-in default source
python3 $P --config docs/formalization-progress/formalization_source.toml
```

`report` reads the source descriptor and writes `FORMALIZATION_PROGRESS.md` and
`formalization_progress.json`. Per-field overrides: `--tag`, `--pdf`, `--text`,
`--title`, `--lean-root`, `--exclude-dir` (repeatable), `--out-md`,
`--out-json`. Precedence: built-in default < `--config` < explicit flags.

### Check for drift

```bash
python3 $P --config docs/formalization-progress/formalization_source.toml --check
```

Exits non-zero (without writing) if the committed outputs differ from a fresh
run. This is what CI runs.

## Continuous integration

`.github/workflows/formalization-progress.yml` runs the `--check` gate on pull
requests and on pushes to `main`. It sets up Python (no poppler; the check
reads the committed extraction) and runs:

```bash
python3 docs/formalization-progress/formalization_progress.py \
  --config docs/formalization-progress/formalization_source.toml --check
```

If the job fails, the committed table is stale relative to the Lean sources, the
paper, or the summaries. **Fix it by regenerating and committing:**

```bash
python3 docs/formalization-progress/formalization_progress.py \
  --config docs/formalization-progress/formalization_source.toml
git add docs/formalization-progress/FORMALIZATION_PROGRESS.md \
        docs/formalization-progress/formalization_progress.json
git commit -m "docs: refresh formalization progress"
```

The check gates rather than auto-commits, so the regenerated table stays under
review. To make a new paper element show as formalized, add a Lean declaration
of the right kind that cites it (e.g. `ŌĆ” per O24 Theorem 5.1 ŌĆ”`) and remove any
`sorry`; the next run reflects it.
